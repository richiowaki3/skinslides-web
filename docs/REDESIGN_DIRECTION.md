# Plan A / Plan B ロジック検証結果と再設計の方向性

作成日: 2026-07-05 / 検証: Claude (Fable 5)
対象:
- Plan A: `index.html` + `player.js` + `logic.js`（v1.10.5）
- Plan B: `audio_trigger_demo.html` + `player.js` + `audio_trigger_logic.js`（v1.10.5）

実データ（`logic_weights.json` 81本 / `sound_metadata.json` トリガー2,534件）に
対する静的分析と再生動態シミュレーションに基づく検証結果。次の実装
アーキテクチャはこの方向性に従って作業すること。

---

## 0. 最重要の構造的問題: 「共通ロジック」は共有ではなくコピペ複製

Plan AとPlan Bは多くのロジックを共有しているように見えるが、実体は
`logic.js` → `audio_trigger_logic.js` への**コピー&ペースト**であり、既に分岐が
始まっている。真に共有されているのは `player.js`（VideoPlayer / プリロード /
フリーズフレーム）のみ。

複製の実態:

| ロジック | 複製箇所数 |
|---|---|
| レベル判定しきい値 (strength 0.25/0.5/0.75 + アタック無条件L4) | **8箇所**（logic.js×3、audio_trigger_logic.js×5） |
| `classifyVideos()` 動画4分類 | 2箇所（内容が既に分岐: logic.jsのみ修正済み） |
| `selectVideoByLevel` / `getScreensToPlay` / `getFibonacciFadeTime` | 各2箇所 |
| `reactScreenWithVideo` / `drawTimeline` / `updateMonitorUI` / `addDecisionLog` / アイドルタイマー | 各2箇所 |

**帰結**: logic.js に適用済みの分類しきい値修正は Plan B には反映されておらず、
Plan B は旧式の偏った分類のまま動いている。今後どちらか一方だけを直すたびに
同じ乖離が再発する。

**方向性（必須）**: レベル判定・分類・選択・振付・タイムライン描画・モニター
表示を `skinslides_core.js`（単一モジュール）に抽出し、Plan A / Plan B 両方が
importする構造にする。しきい値・数式は必ず1箇所にのみ存在させること。

## 1. Plan A の検証結果（確定事実）

### 1-1. 動画選択の極端な偏り
現行 `classifyVideos()`（W/T/H 3値平均、固定しきい値1.0/2.5/4.0）のプールは
L1=5本 / L2=41本 / L3=23本 / L4=12本。対して音響トリガーの実需要は
L1=10.9% / L2=22.4% / L3=21.3% / **L4=45.4%**。

8時間シミュレーション（現行）: **上位10本が全再生の58%を占有**、最頻動画は
207回再生（2.3分に1回）。トリガーの**83%はロック衝突で破棄**（トリガー中央値
間隔0.5秒 vs L1〜L3の動画終了までロック）。

暫定修正済み（logic.jsローカル、未push）: スコアをW/T/S/Hの4値平均に変更し、
しきい値を需要比率から逆算した ≤1.75 / ≤2.25 / ≤2.75 に再較正。
プール 8/18/16/39本 → 上位10本占有率 58%→21%。

### 1-2. アタック無条件L4の過剰格上げ
アタック222件中112件は strength < 0.75 なのに無条件で最上位L4扱い。

### 1-3. 振付ステートマシンの形骸化
L4（全イベントの45%）は固定で3画面全部を取りフロー遷移をバイパス。
ロック飽和と合わせ、1→2→3往復の振付設計はほぼ機能していない。

### 1-4. 反復抑止がトリガー内のみ
除外セットは単一トリガー内のみ有効。直前再生動画の即時再選択を防ぐ機構がない。

### 1-5. 仕様として正しい挙動（変更禁止）
- ロック動画の曲またぎ居座り（トラック終了時にプレイヤーを止めない設計は意図通り）
- L4の1.2秒チャタリング防止ロック
- フリーズフレーム機構（2009年オリジナルの再現要素）
- 方向(L2R/R2L)×フロー方向による90°/270°回転決定

## 2. Plan B の検証結果（確定事実）

### 2-1. 分類の偏りがそのまま持ち込まれ、さらに悪化
`audio_trigger_logic.js` の `classifyVideos()` は旧式のまま（L4=12本）。
デュエット・スライス選択のシミュレーション結果:
- コラージュトリガーのレベル分布: L1=7.2% / L2=18.8% / L3=21.4% / **L4=52.7%**
  （Agent Aが意図的にアタック周辺をスライスするため Plan A より偏る）
- しかも `triggerCollageVideo` は除外セットを渡さない（毎回 `new Set()`）ため
  同一動画の連続出現に抑制が全くない。12本のL4動画が52.7%の頻度で反復される。

### 2-2. コラージュの実効表示時間は約3秒
2エージェント合計で毎秒1.32本の動画が生成され、スロットは4枠 →
**1本あたりの画面滞在時間は約3秒**。動画の平均尺は70秒なので、各動画は冒頭
3秒しか見えずに押し出される。「再生完了で最終フレーム静止(FREEZING)」の実装は
事実上到達不能（評価前に押し出される）。0.4秒のフェードインが滞在時間の13%を
占める。これが意図（高速コラージュ）か要確認。意図なら FREEZING 表示ロジックは
削除、意図でないならスライス内イベントの間引きが必要。

### 2-3. スコア表示のバグ（Plan B固有・要修正）
`updateMonitorUI` のコラージュ分岐:
```js
const wVal = parseFloat(slot.dataset.w) || 3.0;  // ← "0" が 3.0 に化ける
```
LMA値が0の動画（例: `05.mov` = W0/T0/S0/H0）は表示上 W:0 T:0 S:0 H:0 なのに
**Activity 3.00 と表示される**。falsy判定バグ。`Number.isFinite` 判定に修正すること。

### 2-4. スコア表示の不整合（両プラン共通）
- Activity計算がW/T/Hの3値のみでSpace除外（表示はW,T,S,Hの4値を並べる）。
  logic.js側は修正済み、audio_trigger_logic.jsは2箇所とも未修正。
- タイムライン凡例が「Attack(L4)/Swell(L3)/Roll(L2)/Quiet(L1)」と**種別名と
  レベルを混同**。実際の描画色はstrengthしきい値ベースなので、strength0.8の
  「刻み」イベントがオレンジ（Attack）で描かれる。凡例は「L4(強)〜L1(静)」の
  強度表記に改めること。

### 2-5. 死にコード（削除対象）
- `duetState`（ターン交代制の残骸。dad08a7で並行ループ化された際に不要化）
- `MAX_COLLAGE_VIDEOS = 4`（未使用。コメントは「最大6枚」と値とも矛盾）
- `screenIndex`（loadTrackでリセットされるだけで参照なし）
- `triggerEvent` 内のレベル二重計算（tempLvl と audioLevel が同一コードの重複）

### 2-6. 潜在バグ
`videoEl.volume = volume * window.videoGainVolume` — ゲインスライダーが1.0超に
設定されると volume > 1.0 となり DOMException。現UIではダッシュボード非表示の
ため発火しないが、コア抽出時に `Math.min(1.0, ...)` でクランプすること。

### 2-7. リアルタイム解析モードの偏り
`strength: Math.min(1.0, rms * 8.0)` は通常のRMS(≈0.1)で0.8前後に張り付き、
検出ピークのほとんどがL4になる。メタデータモードが既定のため影響は限定的だが、
コア抽出時に正規化を見直すこと。

## 3. 再設計の方向性

### 3-1. 共通コアの抽出（最優先・機械的作業）
`skinslides_core.js` を新設し、以下を一本化:
- `getAudioLevel(event)` — レベル判定（8箇所の複製を全置換）
- `classifyVideos` / `selectVideoByLevel`（logic.jsの修正済み版を正とする）
- `getScreensToPlay` / `getFibonacciFadeTime` / `reactScreenWithVideo` の共通部
- `drawTimeline(canvasId, ...)`（audio_trigger_logic.js版が汎用的なのでこちらを正とする）
- `updateMonitorUI` / `addDecisionLog` / アイドルタイマー

### 3-2. ~~多次元マップ＋スコア関数方式~~ → オノマトペ辞書共通空間方式に変更

**作者判断（2026-07-05）**: 旧案のスコア関数（recencyペナルティ・露出均等化・
連続性項）は**採用しない**。反復は許容する（ただし最終的に全動画が再生されること）。
代わりに、音解析が既に使っている**オノマトペ辞書（質感ベクトル辞書）を共通空間**
として、映像側もそこに射影する方式を採用する。

#### 基盤（すべて既存・追加解析不要）
- 辞書: `Audio analysis data/onomatopoeia_dictionary.csv` — 764語 × 10軸
  (Weight/Time/Space/Flow/Hardness/Moisture/Frequency/Decay/Reynolds/Regularity)
  ＋距離重み (1.0/1.0/0.4/0.5/1.0/0.8/1.0/1.0/0.7/0.6)。
  軸定義と射影式は `onoma_analyzer_v6.py` の DIST_AXES / signature_to_vec が正。
  ※CSVはBOM付きUTF-8。JS側で読む場合はBOM除去必須。
- 音響側: **射影済み**。トラック=`full_vector_0_9`、イベント=オノマトペ語
  （全2,534イベントの語が辞書と100%一致）。イベント語→辞書ベクトルが
  そのまま「今この瞬間の質感ターゲット点」になる。
- 映像側の材料: `metadata_cv2.json`（52本: 運動量・黒接触面積・四肢オレンジ発光数・
  in_side/out_side）＋ `metadata.json`（姿勢遷移・伸展・重心移動・強度）＋
  `logic_weights.json`（人手W/T/S/H）＋ `pauseTime.xml`（フリーズ数）。

#### 映像→10軸射影（初期式、tools/vector_map_prototype.js に実装済み）
Weight←黒接触面積+人手W / Time←運動量+人手T / Space←伸展+人手S /
Flow←重心移動vs.フリーズ / Hardness←最大強度+人手H / Moisture←Hardnessの鏡映 /
Frequency←四肢発光数 / Decay←フリーズ頻度(スタッカート性) /
Reynolds←姿勢乱流+四肢分裂 / Regularity←姿勢変化率。
in_side/out_side はテクスチャ距離には含めず、フレーム連結の振付制約として別途使う。

#### 選択方式（プロトタイプ実証済み・2026-07-05）
イベントのオノマトペ語→辞書ベクトル→各動画との重み付き距離→**距離に応じた
確率選択** P(v) ∝ exp(−dist/T)。決定論的な最近傍K件では同語=同動画に固定され
未使用が出る（K=3で24本、K=10でも11本が未使用）が、確率選択なら全52本が再生される:

| 方式 | top10Share | 未使用 |
|---|---|---|
| 最近傍K=3 | 76% | 24/52 ← 不採用 |
| 確率選択 T=1.5（質感忠実） | 39% | 0/52 |
| 確率選択 T=2.5（推奨初期値） | 31% | 0/52 |
| 確率選択 T=4（分散重視） | 27% | 0/52 |

検証: 動画52本のマップ内ペア距離 median 9.23 ≒ 辞書語彙間 median 8.69
（動画が語彙と同等の広がりで空間に分布＝マップとして機能）。
マップは可読: 例 `13.mp4→うとうと(2.0)`, `11.mp4→ことん/きびきび`,
`05.mp4→おめおめ`。`node tools/vector_map_prototype.js --map` で全52本を確認できる。

#### 映像→音の逆方向（実証済み・2026-07-05, `--v2a`）
同じ空間でトラック/スライスも選べる。表示中の動画群のベクトル重心から、
各トラックの `full_vector_0_9` への距離をランキングし、近い音（継続・調和）／
遠い音（対比・転換）を選ぶ。例: 画面が 13/11/54番のとき、近い音=T522/T538_00/T524、
遠い音=T514/T540/T516。**映像↔音の往復が単一辞書上で成立**（この方式の核心的利点）。
Plan Bのスライス選択も「直前映像の質感語」への距離で置換できる。

#### 射影式のチューニング状況（作者レビューで2軸修正済み）
作者の体感フィードバック（05/46=「ぬー」, 02=「すたすた」, 44=「うわーん/ねとねと」,
08=「じーじー」）を正解データとして射影式を修正:
- **v2: Moisture** を「Hardnessの鏡映」→「黒い接触面積（肌の密着=ねっとり）」へ。
  鏡映式は低Hardness動画30本を湿りコーナーへ固め生成語(ちみゅーん等)に吸着させていた。
- **v3: Frequency(音高)** を映像から測るのを断念し中立固定(4.5)＋距離で無視(wdistVid)。
  映像に音高情報は無く、速度で代用すると「じーじー(きしみ)」と「ぬー(遅い)」のような
  同速・別音高を誤って揃える（実測: 速度案は08.mp4「じーじー」を破壊）。四肢発光数は
  Frequency から Space(空間的広がり) へ移設。
- **効果**: 当てはまり悪(最近傍実在語>6.0)が **21/52 → 9/52** に半減。既存の正解
  (08=じーじー, 13=うとうと, 22=ぎすぎす, 50=ほとほと, 51=くさくさ, 03=とことこ, 44=うわーん/ねとねと)
  は全て維持。05/46は金属音(かーん)を脱し「うーん」(鈍く遅い)へ=「ぬー」の方向。
- **残る ⚠ 9本**: 02,05,07,16,30,44,46,48,56。うち44は語自体は正解(距離のみ⚠)。
  05/46は辞書に「ぬー」語が無く完全一致不能。02「すたすた」はCV解析(伸展0.09=コンパクト)
  と人手H=9が矛盾しており要個別判断。残り(07/16/30/48/56)は作者の体感未確認。
- 射影は tools/vector_map_prototype.js の videoToVec() に集約。作者が --map を見て
  ズレを指摘→該当軸を調整→再実行、を反復する。

### 3-3. トリガー間引き
イベント列をstrengthの局所最大値でダウンサンプリング（2〜4秒窓で最強1件）。
Plan Aではステートマシンを復活させ、Plan Bではコラージュ滞在時間を制御可能にする。

### 3-4. 検証の再現方法
Node.jsで `logic_weights.json` / `sound_metadata.json` を読み、ロック・画面選択・
スライス選択ルールを模擬した8時間シミュレーションを実行。指標は
「上位10本の占有率」「スキップ率」「コラージュ滞在時間」「未使用動画数」。
数式・しきい値を変えたら必ず再実行して比較すること。
